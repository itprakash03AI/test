Welcome to QueryStudio

Your Enterprise Data Platform — Demo Talking Points — April 24, 2026

QueryStudio lets end users connect to any data source (on-prem S3, AWS S3, Oracle, Snowflake, local files), build queries visually, and get results fast. Partition-aware execution on Iceberg tables — 45 min down to 5 sec. Async results persist 7 days. Scheduling, feeds, dashboards, catalog, RBAC, full audit trail. Runs on OpenShift and AWS EKS.

DEMO 1

Connect to Data Sources

5–7 min

On-Prem S3 (Dell ECS / MinIO)

  • Click "New Connection" → select "S3 (On-Premises)"
  • Enter endpoint URL, bucket name, access key, secret key
  • Toggle Path-style access ON — required for Dell ECS and MinIO, without this it won't connect
  • Click "Test Connection" → green checkmark means we can read the bucket
  • Save → connection appears in sidebar, ready to browse
  • "We connect directly to on-prem S3. No gateway, no middleware, no special setup."

AWS S3 (Cloud)

  • Click "New Connection" → select "AWS S3"
  • Enter bucket name, pick region from dropdown
  • Auth: either IAM Role (no keys needed when running in AWS) or access key + secret key
  • Test → Save → same sidebar experience as on-prem
  • "Same interface whether data is on-prem or in AWS. End users don't see the difference."

Local Files

  • Click "New Connection" → "Local Folder" or use the Upload area
  • Drag-and-drop Parquet, CSV, or Excel files — they show up immediately
  • Good for quick one-off analysis or when data hasn't been loaded to S3 yet
  • "For quick analysis, end users just upload a file and start querying. Self-service."

Key point: one platform, three source types, same query experience across all of them.

Partitions & Query Execution

7–10 min

Open an Iceberg Table

  • Click on the S3 connection in sidebar → browse folder structure
  • Click on an iceberg table → columns and partitions get auto-detected
  • "QueryStudio reads the Iceberg metadata automatically. No one had to configure this — it just picks it up."

Explain Partition Pruning

  • Iceberg tables split data into partitions, usually by date
  • 2 years of daily data = roughly 730 partitions, each partition is a separate set of Parquet files on S3
  • Without pruning: query reads ALL 730 partitions — that's ~2.4 TB, takes 45+ minutes
  • With pruning: we only read the partitions the end user selected — say 7 days = 7 partitions, maybe 6 GB, done in 5 seconds
  • Show the partition selector: "Last 7 days", "Last 30 days", "This Month", or manual check/uncheck
  • "This is the single biggest performance win. Same data, same question — 45 minutes vs 5 seconds. It's automatic."

Build & Run a Query

  • Pick columns from the column picker (drag or checkbox)
  • Add a filter: e.g., amount > 1000
  • Select partitions: "Last 30 days"
  • Point out the SQL preview panel — "The SQL is always visible. End users don't need to write it, but they can see exactly what's running."
  • Click "Run Query" → show real-time progress bar (updates via WebSocket)
  • Results appear in data grid — point out row count, execution time, partitions scanned vs total
  • "Completed in X seconds, scanned Y partitions out of Z total. That's the pruning at work."

Key point: end users just pick a date range. The platform handles which files to read and which to skip.

Transition: "Now let's save this query so we can reuse it..."

Save & Reuse Queries

2–3 min

Save

  • Click "Save Query" → enter name (e.g., "GL Transactions — Last 30 Days") and description
  • This saves the full configuration: tables, columns, filters, partitions, joins — not just the SQL text
  • "The entire query setup is preserved. Next time you open it, everything is exactly where you left it."

Reload

  • Open "Saved Queries" in sidebar → click a saved query → full state restores in query builder
  • Modify if needed and re-run — no rebuilding from scratch

Beyond Saving (mention, don't demo)

  • Publish — saved queries can become read-only "Published Views" that other teams consume as data endpoints
  • Schedule — set up daily or weekly automated execution
  • Feed source — use as input for automated data delivery to S3, Oracle, or SFTP

Key point: build once, reuse forever. Queries become reusable assets, not throwaway work.

DEMO 4

Submit & Walk Away

3–5 min

Submit a Large Query

  • Pick a bigger dataset (more partitions, more rows) and click "Run Query"
  • Show the progress bar with percentage and row count updating in real time
  • "This is running on the server. I don't need to keep the browser open."

Close & Come Back

  • Close the browser tab — "Let me close this and come back later."
  • Re-open QueryStudio → navigate to Execution History
  • The query continued running on the server. Results are persisted for 7 days.

Execution History

  • Show the list: Green = Completed, Yellow = Running, Red = Failed (with error message)
  • Click a completed execution → results load in data grid
  • Shows: query name, status, row count, duration, timestamp — full trail of what ran
  • "Complete visibility. You can see what ran, when, how long, how many rows. Audit trail for everything."

Download

  • Download button → CSV. Export dropdown → Excel, JSON.
  • "One click to get data out in whatever format the team needs."

Key point: end users don't wait around. Submit, walk away, results are there when they come back.
